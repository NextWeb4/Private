本地文档搜索客户端：一次离线工具的工程化复盘

2026-06-27 技术与网络

来源：2026-06-27 开发本地文档搜索客户端会话

这篇文章由本地 Codex 历史会话整理而来，保留可复用的工程经验，省略私有路径、账号、临时日志和不适合公开发布的细节。

目标：能搜，还要能证明它真的可用

这次本地文档搜索客户端的会话不是简单做一个搜索框，而是围绕“离线资料如何被稳定检索”展开。项目里同时关注索引、筛选、预览、范围控制、文件类型过滤和 UI 卡顿感，最后还补了 UI 自动化证据。

关键判断是：搜索工具不能只靠手动点几下说能用。它需要有可重复运行的构建、测试、UI smoke、扫描和清理流程，才能证明后续迭代没有破坏主路径。

做法

先给主窗口和关键控件加稳定的 AutomationId：搜索框、模式筛选、类型筛选、范围筛选、结果表、预览区和状态栏。这样 UI 自动化不依赖脆弱的坐标点击，也不会因为布局调整就失效。

再增加 UIA smoke runner 和 PowerShell 启动脚本，把“输入关键词、等待结果、检查预览和状态”变成可运行检查。相比截图确认，这类测试更能覆盖真实交互。

最后把测试结果写进报告，并做静态扫描：占位符不应残留，网络行为只应出现在明确的阻断模块和对应测试里，dist 目录只保留正式 EXE。

经验

离线工具的质量边界包括两部分：功能正确，以及没有偷偷越过离线边界。后者同样需要扫描和文档记录。

UI 自动化不是大型项目才需要。只要工具有稳定主路径，给关键控件命名并写 smoke test，就能显著降低后续修改的回归成本。

“不卡顿”不能只凭感觉。至少要在构建、单元测试、UI 冒烟和人工体验之间形成一组互相补充的证据。
